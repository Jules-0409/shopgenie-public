# ENGINEERING_RULES.md — 通用工程规则（可移植）

> **这是一份项目无关的通用规则,适用于任何项目。** 新项目直接复制本文件到仓库根目录;
> 在项目的 `AGENTS.md` / `CLAUDE.md` 里写一句"通用规则见 ENGINEERING_RULES.md",再补本项目专属内容(契约、模块图、路线图)。
>
> 规则分三级:**[MUST]** 违反即视为缺陷,不得合入;**[SHOULD]** 默认遵守,偏离需在提交信息说明理由;**[GUIDE]** 建议。

---

## 1. 代码正确性护栏 ★（给"看不懂代码"的 owner 的保险）

> 前提:项目 owner 可能无法亲自审阅代码逻辑。那么"代码是否正确"这件事,不能靠 owner 肉眼把关,
> 必须靠**可执行的证据 + 强制流程**来保证。本章是整份规则里最重要的一章。

1. [MUST] **行为改动必须有测试证明。** 不存在"如果这处逻辑写错了、会失败的测试",就不算完成。测试是 owner 看不懂代码时唯一靠得住的裁判。
2. [MUST] **测试要覆盖失败路径和边界,不止 happy path。** owner 没法替你想到漏掉的边界,所以这条强制你自己想全:空输入、超限、并发、失败重试、越界。
3. [MUST] **改完必须自证,拿出证据。** 实际跑一遍(测试输出 / 运行截图 / 命令结果)贴出来,不能只说"应该没问题 / 我觉得可以"。owner 信的是证据,不是保证。
4. [MUST] **重构 = 行为不变 + 测试守护;真改了行为必须显式点明。** 重构后测试应原样通过(证明没改坏)。任何**行为变化**必须在提交信息和进度文件里用**大白话**标注"⚠️ 这里行为变了:原来 X,现在 Y,因为 Z",绝不偷偷改。
5. [MUST] **不静默吞错。** 输入在边界处校验;出错就响(记录或抛出),禁止 `except: pass` 这类把错误藏起来的写法——藏起来的错误 owner 永远发现不了。
6. [MUST] **一切循环 / 递归 / 资源都要有上限。** 防止失控(死循环、无限递归、内存爆炸)。每个上限要有配置项和触发时的明确行为。
7. [MUST] **每个非平凡改动配一段大白话说明:做了什么、为什么、怎么验证的。** 让 owner 即使不读代码,也能判断"意图对不对"。看不懂 `for` 循环没关系,但要能看懂"这段是用来防止重复扣款的"。
8. [SHOULD] **不确定就停下来问,别在含糊的逻辑上瞎猜。** 涉及钱、数据删除、权限、对外发送等高后果逻辑,宁可多问一句。
9. [SHOULD] **复杂逻辑就近写注释解释"为什么这么做"**,而不只是"做了什么"。半年后(或换个 agent)还能看懂。

## 2. 契约纪律（防止跨模块 / 跨端漂移）

1. [MUST] **数据契约有唯一源头。** 后端模型 / schema 是真相源,前端类型、其他语言的镜像必须与之逐字段对齐。
2. [MUST] **改契约必须同帧同步所有镜像,并加测试。** 同一个提交里改完所有相关端,提交信息打 `[contract]` 提醒。否则今天能跑、明天某处悄悄对不上。
3. [MUST] **枚举集中定义,不散落字符串字面量。** 新增一个枚举值,要同步所有用到它的地方(后端、前端、渲染分支),缺一不可。

## 3. 模块与文件结构（按职责/功能拆分）

1. [SHOULD] **单文件聚焦,防止上帝文件。** 源文件 soft cap **~400 行**、函数 ~50 行;逼近上限就拆。**一个文件 = 一个清晰职责。** 新增大块逻辑优先放新模块,而非往已大的文件里堆。
2. [MUST] **分层依赖单向,下层不 import 上层。** 例如 `存储层` 不得 import `业务层`;`工具层` 不得知道 `调度层`。
3. [SHOULD] **按"功能/职责"切分,不按"文件类型"堆。** 一个功能(记忆、沙箱、规划…)的相关代码聚在一个模块/目录里。判断标准:**改一个功能,理想情况只动一个目录。**
4. [GUIDE] **怎么拆 —— 两种常用手法:**
   - **拆成包(目录)**:一个文件大了,按子职责拆成多个文件,用 `__init__` 统一 re-export,**对外 import 不变**(外部无感知)。例:`runtime.py` → `runtime/`(events / base / lifecycle / planning / execution)。
   - **拆有状态的大类(mixin/组合)**:一个类太胖,按职责拆成多个 mixin,各放一个文件,主类多继承组合;共享状态放一个 base 类。例:`TaskRunner` = 生命周期 + 规划 + 执行 三个 mixin 组合。
5. [GUIDE] **拆分是纯重构:行为必须不变、测试原样通过**(对照 §1.4)。拆完跑测试证明没改坏,才算完成。

## 4. 安全基线

1. [MUST] **不可信输入 / 代码执行必须沙箱化:** 资源(CPU/内存/超时)、路径白名单、网络、危险操作黑名单,多重限制。
2. [MUST] **绝不提交密钥。** `.env` 等入 `.gitignore`;每次提交前核验密钥没被 staged。配置用占位符 + 环境变量。
3. [MUST] **高后果操作(删除/转账/对外发送/改权限)默认需要确认**,不擅自执行。
4. [MUST] 不为了"让 demo 跑通"临时关掉安全限制后忘了恢复;如需临时放宽,记录并设回滚项。

## 5. 测试与"完成"的定义（DoD）

一个改动算"完成",必须同时满足:
1. 全部测试通过。
2. 类型检查 / 构建无错。
3. 改了契约 → 已同帧同步 + 加了对应测试(§2)。
4. 改了核心逻辑 / 安全 → 有覆盖该路径的测试,不能只手测(§1.1)。
5. 关键可观测数据(耗时 / 成本 / 用量等)在新路径上已记录。
6. 进度文件已更新(§6)。

## 6. 进度同步（PROGRESS 文件)

每个项目维护一份**进度文件**(如 `docs/PROGRESS.md`),它是"项目现在做到哪"的**唯一权威来源**。

1. [MUST] **完成或卡住一项,立即更新进度文件**,作为该项"完成"的一部分(见 §5 DoD)。需求层面的变化同步需求文档。
2. [MUST] **进度文件必须始终反映真实进度** —— 任何人 / 任何 agent 接手,只看它 + `git log` 就能续上,不用翻聊天记录。
3. [SHOULD] **推荐结构**(让接手者 30 秒看懂):
   - **当前状态**:一句话快照(哪些阶段已完成 / 进行中 / 下一步)+ 测试数。
   - **历史日志**:按阶段/任务分条,每条写清「做了什么 / 关键决策 / 怎么验证的 / 遗留」。
   - **遗留与待办**:明确列出未完成项及其阻塞原因(缺依赖?待环境?),别让它们只活在记忆里。
4. [SHOULD] 进度文件**只记进度**,不重复规则(DoD/红线/协作纪律以规则文件为准),避免两处漂移。
5. [GUIDE] 诚实优先:**做不完、没验证的,如实标注 ⚠️**,不要写成"已完成"。进度文件骗人 = 比没有更糟。

## 7. 版本控制与协作纪律

1. [MUST] **动手前先 `git init`**,任何实质改动都在版本控制下。
2. [MUST] **小步提交、语义清晰**,一个提交一件逻辑事。契约改动打 `[contract]`。
3. [MUST] **功能在专门分支上做**,不直接在主干上改;**绝不 push 到主干/远程,除非 owner 明确同意**。
4. [MUST] **不删除别人的测试 / mock 来让自己的改动"看起来通过"。**
5. [SHOULD] 不并发改同一文件(多 agent 协作时按模块分工或串行)。
6. [SHOULD] 推翻既定决策 / 放宽安全 / 重大架构选择 → 记进决策日志(日期 + 原因)。

## 8. 编码通则

1. [MUST] 全量类型标注(静态类型语言),边界处做数据校验。
2. [SHOULD] I/O(网络 / 磁盘 / DB)异步优先,不阻塞主循环;CPU 密集放线程/进程池。
3. [SHOULD] 函数短、单一职责;复杂逻辑下沉到合适的层。
4. [GUIDE] 配置走配置文件 + 环境变量,不硬编码常量。

---

### 一句话总览
> 因为 owner 看不懂代码,所以**每一处逻辑的正确性都必须由"会失败的测试"和"跑出来的证据"来背书**,而不是由"我觉得没问题"来背书。改之前先想:我有没有破坏契约、绕过安全、漏写测试、悄悄改了行为而没说?
